Webbplatsens design

För att kunna fokusera fullt ut på syftet med uppgiften - att tagga upp en komplett webbplats med korrekt HTML- och CSS-kod – har jag valt att använda mallen för sidans design som tillhandahålls i kursen.

Desktop first

Ett av de första ställningstagandena man behöver göra är om man ska närma sig projektet ”desktop first” eller ”mobile first”. Efter att ha tagit del av en genomgång av för- och nackdelar (Johnson, J. 2019)2 valde jag ”desktop first” eftersom jag håller med om att det är roligare att ha hela paletten av möjligheter till sitt förfogande från början och sedan skala ner mot mindre skärmar. Det är också så som ämnet har angripits inom kursen hittills i demonstrationer och föreläsningar så det känns vedertaget.

CSS reset

Jag väljer att använda CSS reset från Meyerweb (Meyer, E., 2011)1 därför att den tar bort all formatering, något som Parik Bernhardsson också nämner i sin föreläsning. Det passar mig bäst då jag tycker om att starta med ett blankt blad och vill slippa lägga tid på att undersöka varför en viss regel inte slår igenom som den ska.

Font size

Det finns många sätt att bestämma storleken på typsnitt på webbplatsen men jag valde att följa riktlinjer från en guide som heter Shaping Design (EditorX 2021)3. Där föreslås att man sätter 10px som font-size i HTML-taggen för att sedan använda rem i de övriga CSS-reglerna och få jämt delbart med 10. Det tycker jag känns väldigt tydligt och enkelt.

Grid eller Flexbox?

Jag lade mycket tid på att reflektera över och undersöka för- och nackdelar med att använda grid som struktur för huvudsidan. Det kändes som en god idé att göra det men samtidigt var det något som tog emot. Jag hittade en genomgång av skillnader mellan flexbox och grid på MDN web docs (MDN, 2023)5 som jag hade stor nytta av.

Jag har från början haft en förkärlek för flexbox och jag tror jag fick en förklaring i den här artikeln, nämligen att flexbox lämpar sig när man vill anpassa layouten efter innehållet. När jag tittade på mallen för designen började jag fundera på hur jag skulle få innehållet att se likadant ut i min kod. Då är det lätt att tänka område för område – header, navigation, huvudinnehåll, footer – och designa varje område med flexbox. Det var först vid Christian Olssons spontana genomgång av grid vid ett klasshäng i zoom som jag fick klart för mig att man kan kombinera grid och flexbox på ett för mig idealiskt sätt. Jag kunde använda grid som struktur för hela sidan och sedan använda flexbox inom de olika områdena för att få innehållet så som jag ville ha det. Det gjorde att jag testade att lägga allt innehåll i ett grid och det blev en bra lösning.

Grid-template-area

Eftersom jag har lättare att jobba visuellt än matematiskt så föredrar jag att använda grid-template-areas istället för att lägga element från en grid line till en annan. Jag har använt metoden som beskrivs i Smashing Magazine (Andrew, R. 2020)6 för att placera innehållet på förstasidan enligt mallen. Det blir lite mer avstånd mellan kolumnerna än i mock-upen men jag tycker det blir snyggare och ger ett luftigare intryck när det är så pass mycket text.

Tillgänglighet

Till navigeringen har jag använt en oordnad lista eftersom det rekommenderas av tillgänglighetsskäl (W3C 2017)4.

Jag debatterade med mig själv om jag skulle lägga huvudinnehållet på förstasidan i ett main-element eller inte. Rent tekniskt verkade det fungera lika bra att lägga de tre kolumnerna som div-element direkt under wrapper. Men jag gick tillbaka igen och läste mer om semantiken och bestämde mig för att ändå lägga till main-taggar runt div:arna framför allt för tillgänglighetens skull. Main-taggen förbättrar förståelsen för innehållet så att skärmläsare kan skippa återkommande element, såsom menyer, för att komma direkt till huvudinnehållet (MDN 2023)10.

Negativa marginaler

För att åstadkomma layouten med de runda fotona som överlappar elementen som kommer före och efter så har jag använt negativa marginaler. Åsikterna verkar gå isär om huruvida detta är ett ok sätt att använda negativa marginaler och jag hittar ingen ”officiell” hållning. Men jag finner stöd för metoden (Zhang, S. 2022)9 om det är för att positionera element där box-modellen sätter gränser och man vill ha ett alternativ till absolute positionering. En nackdel med att använda negativa marginaler kan vara att det blir rörigt i koden, så jag har lagt in en kommentar vid varje tillfälle för att förbättra tydligheten.

Media queries screen

Den första brytpunkten kommer vid 960px där jag tyckte att sidan började ”gå sönder”. Navigationstexten kommer in över det gula i header-bakgrunden och blir svår att läsa så då fick den en mörkare bakgrund. Jag gick över till en layout med bilderna till vänster och text till höger i huvudinnehållet för att bli av med de långa textspalterna som uppstår när man har dem sida vid sida. Läsaren ogillar att skrolla (Bergström, B. 2021)8 så det är bättre att försöka få in texten på en skärmsida. Samtidigt passade jag på att ändra ordning på info-boxarna med hjälp av css order så att de ligger i den ordning de ska ha i mobilversionen.

Den layouten fungerar ner till ungefär 600px då menyn snart sträcker sig över hela bredden och jag väljer att lägga in en hamburger-meny i stället.

Media queries print

Det står i beskrivningen av uppgiften att man kan lägga media queries för print i en separat CSS-fil. Vi pratade också om det på ett klasshäng och det verkar vara dels en smaksak och dels en fråga om hur mycket kod man har. Jag funderade vidare på det och sökte för- och nackdelar på nätet men hittade inget som var helt klockrent. Enligt CSS-Tricks (Coyier, C. 2009)7 kan det finnas skäl att lägga CSS-kod som är unik för en viss webbsida i en egen CSS-fil, men jag hittar inga bra argument för att lägga just media queries för print i en egen fil så jag har lagt dem längst ner i style.css.

För att göra innehållet mer läsvänligt i utskriftsversionen har jag ändrat layouten från grid och flexbox till block (Buckler, C. 2022)11. Det tar bort all fancy layout så att innehållet skrivs ut på ett enklare manér.

Rapportsidan

Rapporten med runt 1200 ord som ska in på samma sida skapar sitt eget lilla bekymmer. Här behöver läsbarheten verkligen tas med i beräkningen och innehållet delas upp i lagom stora bitar för att presenteras på ett attraktivt och lättläst sätt. Efter en del research har jag valt att använda kantrubriker i desktop-varianten, vilket är en god idé för en lite bredare skärm (Bergström, B. 2021)12. Kortare textstycken har jag lagt i något som kan likna faktarutor med avvikande bakgrundsfärg för att skapa kontrast. På mindre skärmar presenteras texten i en kolumn med tydliga rubriker och jag behåller faktarutorna med bakgrundsfärg.
